脱壳

加固

对APK加固的方式一般是下面四种，从上到下加固强度逐渐变强，同时对性能的影响也逐渐变大：

  1. 代码混淆
  2. 动态加载dex
  3. 函数虚拟化
  4. VMP指令虚拟化

代码混淆

  1. 编译的时候，将源代码中的变量名、函数名、类名替换成没有语义的单词或者字母。比如String name = 'Alex';就会变成String a = 'Alex';。同时，还会加入一些垃圾代码，最显著的就是修改流程控制语句：原本两个分支，肯能会改成四五个分支，一个循环嵌套另一个循环，两个函数之间的直接引用可能在中间会被跳好几层。加上代码量有成千上万行，使得人工静态分析的难度变大，这就是直接看源码和看反编译出来的源码的区别。

  2. 这种加固方式对静态分析来说，还算比较友好，市面上有这种反混淆的工具。比如，jadx-gui中就自带了反混淆的功能，只需要在反编译之前打开，就能在一定程度上将名称还原成易于人类阅读的单词。

动态加载dex

  1. 如果不做任何措施或者说只进行了混淆，那么我们把APK解压缩之后就可以直接获得源码文件。编译的时候可以把编译生成的“.dex”文件，进行加密然后存入到“.so”文件中，并在“.so”文件中提供一个入口函数，用来解密 -> 执行 -> 返回结果。
  2. 那么，谁来调用被隐藏的”.dex”原字节码呢——壳程序（shell）。壳程序被打包在APK里面，运行该APP时，系统真正进入执行的是壳程序，壳程序的功能就是调用原程序。
  3. 有些壳程序会把解密之后的“.dex”文件（部分或者全部）存放到系统磁盘中，方便下一次快速运行，提高程序性能。但还有部分“.dex”文件就只会在运行的时候被加载进内存中，不会被保存到本地（熟悉web安全的同学，应该比较清楚内存马就是这个原理）。
  4. “.dex”被动态加载之后，就无法通过反编译直接看到源代码了，因为反编译就是把“.dex” -> “samli” -> java/kotlin，现在“.dex”都被藏进“.so”，而且还被加密了，自然就不能直接拿到源码了。
  5. 因为多了壳程序的中间调用（解密、动态加载），所以性能一般来说是比代码混淆降低的多。

函数虚拟化

  1. 把隐藏、加密的“.dex”找出来解密就能拿到比较完整的源码了。函数虚拟化在动态加载的基础之上，做了“颗粒度”更小的动态加载：把函数体抽取出来，加密存入“.so”，在原函数体中调用native方法，动态解密加载函数体，而且一般来说是不会有本地文件的产生来存储原函数体的。
  2. 性能比动态加载dex更低。

VMP指令虚拟化

  1. 除非你的目的是就是一定要逆向这个APK，否则建议直接放弃。

  2. java源代码到安卓系统中的运行流程：Java —javac—> .class —d8/r8—> .dex —ART—> 汇编 ——> 二进制 ——> CPU。VPM虚拟化：

  3. VMP虚拟化，简单理解就是在函数虚拟化的基础之上，进一步对函数体对应的指令抽取出来，按照自定义的规则进行替换，最终编译成一个自定义的指令文件，并加密存储到“.so”文件中。最终，安卓系统调用这个函数时，通过native层的代码将函数体解密并加载进内存中，并由内存中的VMP解释器（APK包中会带有这个解释器，一般是放在“.so”中）转为汇编、二进制，最后交给CPU。

  4. 在这四种加固方式中，VMP指令虚拟化的方式是最安全的，对程序运行性能的影响也是最大的，一般只会对核心的函数（比如支付、人脸认证等）才会进行处理。

查壳

APK就是一个压缩包，解压之后就能看到很多代码文件以及二进制文件。查壳的原理就是将apk解压之后，读取里面所有”.so”类型的文件名，根据文件名判断对应的壳类型，与cms指纹识别是一样的道理，关键在于指纹库的强大。

  1. 自己写一个脚本（ApkScan-PKID早就不更新了，且指纹太老，不建议使用）

  2. Detect It Easy（DIE）：开源免费，且一直更新，可通过Microsoft Store下载安装。功能比较强大，里面整合了多个工具的功能。

  3. Virus Total：专门分析恶意程序的平台，可以用来分析apk，解析相关信息。DIE工具里面就内置了这个平台

  4. ApkCheckPack：根据“.so”的文件名和路径执行指纹进行匹配，还能检测反root、反调试、反模拟器采用的手段，以及检测apk中的硬编码信息。https://github.com/moyuwa/ApkCheckPack/releases

检测frida

检测手段

  1. 遍历运行的进程列表从而检查frida-server是否在运行
  2. 遍历data/local/tmp目录查看有无frida相关文件
  3. 遍历映射库，检测frida相关so
  4. 检查27042这个默认端口
  5. 内存中扫描frida特征字符串 “LIBFRIDA”
  6. frida使用D-Bus协议通信，我们为每个开放的端口发送D-Bus的认证消息
  7. 检测有无frida特征的线程名（如gum-js-loop）
  8. 检测库函数开头八个字节，判断是否被frida hook

绕过检测

  1. 修改默认端口

  2. frida-server和android_server的名字不要直接写本名，改成as，fs这种

  3. server位置不要放在data/local/tmp,可以放在data/目录下

对抗反root（内核root）

Apatch

  1. 为什么要进行内核root

  2. 确保手机已经解开BL锁、用户态已经取得root权限、电脑准备好adb和fastboot（fastboot devices检测不到设备是因为电脑没装驱动，最后面有WriteUp）

  3. 提取当前的boot.img文件到电脑中

  4. 从官网或者第三方刷机平台（如果我上传ROM包到第三方平台，我会在内核植入免杀马，所以谨慎下载！！！），下载当前系统的ROM包，建议手机型号和内核和原系统保持一致。Nexus和Pixel手机可参考：https://developers.google.cn/android/images?hl=zh-cn

  5. 得到的是一个压缩包，把它解开，里面还有一个image-xxxxx.zip的压缩包，继续解开，一会要对里面的boot.img进行魔改。通过adb将boot.img上传到手机（比如：adb push boot.img /sdcard/Download）

  6. 下载APatch.apk到手机：https://github.com/bmax121/APatch/releases

  7. 运行apatch，选择刚刚上传的boot.img，进行修改

  8. 将apatch打入内核，并设置密码

  9. 修改完成之后，屏幕中日志会告诉你修改之后的boot.img的位置，通过adb将它pull到电脑上

  10. 手机进fastboot模式（adb reboot bootloader），将修改之后的boot.img刷入boot分区

  11. 重启之后，apatch正常运行，各个模块没有提示兼容性问题

  12. fastboot找不到设备

    • 电脑打开设备管理器，在其他设备或者便携设备中找到你的设备名称（如果没有，可以点击最右侧的“更多操作” ==> “查看” ==> “勾选显示隐藏的设备”）

    • 双击你的手机，点击更新驱动程序

    • 如果说fastboot仍然找不到设备，或者说根本没有Android Phone这个选项，手动下载驱动：https://gitcode.com/open-source-toolkit/1de04/blob/main/Fastboot%20%E9%A9%B1%E5%8A%A8.zip，然后解压

    • 如果你的系统存在Android Phone这个选项（右键 → 更新驱动 → 浏览我的计算机 → 让我从计算机的设备驱动列表选择） ==> 仍然双击进来 ==> 点击“从磁盘安装” ==> 选择刚刚解压得到的驱动文件（android_winusb.inf） ==> 选中后确定 ==> 下一步 ==> 弹出警告选「始终安装」

    • 如果系统中不存在Android Phone，那么选择移动设备（右键 → 更新驱动 → 浏览我的计算机 → 让我从计算机的设备驱动列表选择），然后重复上述步骤

    • 提示「第三方驱动无签名」导致装不上，先临时关掉驱动签名校验：Win10 设置 → 更新和安全 → 恢复 → 高级启动 → 立即重启 → 疑难解答 → 启动设置 → 重启 → 按7选「禁用驱动程序强制签名」

安装痕迹隐藏模块

  1. cherish_peekaboo.kpm

    • 在APatch内核里直接hook掉kallsyms、show_module、seq_puts等函数，屏蔽掉 apatch、supercall、ksu等关键字
    • 下载插件并上传到手机https://archive.org/download/cherish_peekaboo_kpm_apatch
    • APatch主界面 → 内核模块 → 右下角“+” → 选cherish_peekaboo_1.5.kpm → 开始修补 → 成功后重启

  2. ZygiskNext

    • 给后面所有Zygisk模块提供注入环境，本身也带一点隐藏Zygote痕迹的功能
    • 现在插件压缩包并上传到手机（不要解压）https://github.com/Dr-TSNG/ZygiskNext/releases
    • 打开APatch Manager → 底部栏点 “系统模块” → 右下角 “+” → 文件管理器里选中刚才的ZygiskNext-xxx.zip
    • 安装完成后界面会出现 “ZygiskNext” 卡片，右侧开关默认已打开 → 直接点右下角 “重启”。
    • 重启回来再进 APatch → 系统模块，能看到 ZygiskNext 卡片里出现 “已启用” 字样即表示运行成功

  3. TrickyStore

    • 伪造keystore和verifiedBootState，安全措施比较严格的软件（比如金融类的、交易类的）会检测这一块
    • 下载插件并上传到手机https://github.com/5ec1cff/TrickyStore/releases/，安装步骤同ZygiskNext
    • 重启之后，检查是否存在/data/adb/tricky_store/keybox.xml和/data/adb/tricky_store/target.txt
    • 如果没有，准备1份未吊销的keybox.xml（可选，但想拿STRONG完整性必须放，注意不可以自动生成）。可以从海鲜市场购买或者从其他手机adb pull /data/misc/keystore，然后上传到手机
    • 编辑targets.txt，把要生效的App包名逐行写进去，若手机TEE已损坏，在包名后加英文感叹号!强制走软证书模式
    • 保存退出，TrickyStore 实时热加载，立刻生效，不需要重启

  4. Zygisk-Assistant

    • 删除su二进制残留、清空libcvm调用栈、隐藏Zygisk自身特征

    • 下载插件https://github.com/snake-4/Zygisk-Assistant/releases/，安装步骤同ZygiskNext（注意，确保ZygiskNext插件中的排除列表策略处于关闭状态）

    • 在超级用户里面，点击某个app，然后打开排除修改，Zygisk-Assistant插件就能对该app进行特征隐藏
